# Redis 常见面试题及答案

## 一、核心概念与数据结构

### 1. Redis 是什么？有哪些应用场景？
- **定义**：Redis（Remote Dictionary Server）是一个开源的、基于内存的键值存储数据库，支持多种数据结构，可用作数据库、缓存和消息中间件。
- **核心特性**：高性能、支持丰富的数据结构、持久化、主从复制、事务、发布/订阅、Lua脚本、高可用（哨兵）和分布式（集群）。
- **应用场景**：
  - **缓存**：减轻数据库压力，提升系统响应速度
  - **会话存储**：分布式Session共享
  - **排行榜/计数器**：利用ZSET、INCR等命令
  - **消息队列**：基于List的阻塞队列、Stream类型
  - **实时系统**：如用户在线状态、实时排行榜
  - **分布式锁**：通过SETNX、RedLock等实现
  - **地理位置**：使用GEO相关命令

### 2. Redis 支持哪些数据类型？各自的使用场景是什么？

| 数据类型 | 底层实现 | 特点 | 应用场景 |
|---------|---------|------|---------|
| **String** | SDS（Simple Dynamic String） | 二进制安全，可包含任何数据 | 缓存、计数器、分布式锁、Session存储 |
| **Hash** | 压缩列表/哈希表 | 键值对集合，适合存储对象 | 存储用户信息、商品属性等结构化数据 |
| **List** | 压缩列表/双向链表 | 按插入顺序排序，可重复元素 | 消息队列、最新文章列表、关注列表 |
| **Set** | 整数集合/哈希表 | 无序集合，元素不重复，支持集合运算 | 标签、共同好友、随机抽奖 |
| **ZSet** | 压缩列表/跳表+字典 | 有序集合，每个元素关联一个分数 | 排行榜、延时队列、带权重的任务队列 |
| **Bitmaps** | String的位操作 | 位操作，节省空间 | 用户签到、活跃用户统计 |
| **HyperLogLog** | 特殊数据结构 | 基数估算，误差率约0.81% | 独立IP统计、UV统计 |
| **GEO** | ZSET扩展 | 地理位置信息存储与计算 | 附近的人、地点搜索 |
| **Stream** | 基数树+链表 | 消息流，支持消费者组 | 消息队列、事件溯源 |

### 3. Redis 为什么快？（高性能原理）
1. **基于内存操作**：数据存储在内存中，读写操作直接操作内存，速度比磁盘高几个数量级
2. **单线程模型**：避免了多线程上下文切换和竞争条件，同时保证了原子性操作
3. **I/O多路复用**：使用epoll/kqueue等系统调用，单线程处理大量并发连接
4. **高效的数据结构**：
   - 自定义的SDS字符串，预分配空间，O(1)获取长度
   - 压缩列表、跳表等空间和时间高效的数据结构
5. **优化措施**：
   - 渐进式rehash
   - 写时复制
   - 惰性删除

> **注意**：Redis 6.0引入了多线程I/O（默认关闭），但命令执行仍然是单线程，多线程只用于网络数据的读写，进一步提升高并发场景下的性能。

## 二、持久化与内存管理

### 4. Redis 的持久化机制有哪些？有什么区别？

**RDB（Redis Database）**
- **原理**：在指定时间间隔内，生成数据集的时间点快照
- **触发方式**：
  - 手动执行`SAVE`（阻塞）或`BGSAVE`（后台异步）
  - 配置文件中的`save`规则
  - 主从复制时，主节点自动触发
- **优点**：
  - 紧凑的二进制文件，适合备份和灾难恢复
  - 恢复大数据集时速度比AOF快
  - 最大化Redis性能（父进程无需磁盘I/O）
- **缺点**：
  - 可能丢失最后一次快照后的数据
  - 数据集大时，fork子进程可能阻塞主进程

**AOF（Append Only File）**
- **原理**：记录每个写操作命令，以追加方式写入文件
- **同步策略**：
  - `appendfsync always`：每个命令都同步，数据最安全，性能最低
  - `appendfsync everysec`：每秒同步一次，推荐配置
  - `appendfsync no`：由操作系统决定，性能最好，可能丢失数据
- **优点**：
  - 数据更安全，最多丢失1秒数据
  - AOF文件易于理解和解析
  - 提供rewrite机制压缩文件
- **缺点**：
  - 文件通常比RDB大
  - 恢复速度较慢
  - 对性能有一定影响

**混合持久化（Redis 4.0+）**
- 结合RDB和AOF的优点
- 开启`aof-use-rdb-preamble yes`后，AOF文件由两部分组成：
  - RDB格式的全量数据
  - 增量AOF日志
- 兼顾了启动速度和数据安全性

### 5. Redis 的内存淘汰策略有哪些？
当内存使用达到`maxmemory`限制时，Redis会根据配置的策略淘汰数据：

1. **noeviction**（默认）：不淘汰，返回错误
2. **allkeys-lru**：从所有key中使用LRU算法淘汰
3. **volatile-lru**：从设置了过期时间的key中使用LRU算法淘汰
4. **allkeys-random**：从所有key中随机淘汰
5. **volatile-random**：从设置了过期时间的key中随机淘汰
6. **volatile-ttl**：淘汰过期时间最小的key
7. **allkeys-lfu**（4.0+）：从所有key中使用LFU算法淘汰
8. **volatile-lfu**（4.0+）：从设置了过期时间的key中使用LFU算法淘汰

**建议**：通常推荐`allkeys-lru`，除非有明确的需求。

### 6. 如何优化 Redis 内存使用？
1. **使用合适的数据类型**：
   - 小数据用ziplist，大数据用hashtable
   - 使用hash存储对象而非多个string
2. **控制key的数量**：
   - 避免大量小key
   - 使用hash tag确保相关数据在同一slot
3. **配置优化**：
   - 设置合理的maxmemory和淘汰策略
   - 使用`hash-max-ziplist-entries`等参数控制编码转换阈值
4. **内存压缩**：
   - 开启`activerehashing`减少内存碎片
   - 使用`memory purge`手动清理内存
5. **监控与预警**：
   - 监控`used_memory`、`mem_fragmentation_ratio`等指标
   - 设置内存使用率阈值告警

## 三、高可用与集群

### 7. Redis 的主从复制原理是什么？
**复制流程**：
1. **建立连接**：从节点执行`SLAVEOF`命令，向主节点发起连接
2. **同步初始化**：
   - 主节点执行`BGSAVE`生成RDB文件
   - 传输RDB文件到从节点
   - 从节点清空旧数据，加载RDB
3. **命令传播**：
   - 主节点将写命令发送到从节点
   - 从节点执行命令保持数据同步
4. **增量同步**：从节点断线重连后，会尝试部分重同步，只同步断开期间的数据

**复制模式**：
- **全量同步**：从节点初次连接或runid变化时触发
- **部分同步**：基于复制积压缓冲区，从节点断线时间短时可以只同步缺失数据
- **无盘复制**：主节点不生成RDB文件，直接通过网络传输数据

### 8. Redis Sentinel（哨兵）的工作原理？
**作用**：Redis的高可用解决方案，提供自动故障转移、监控和通知功能。

**工作原理**：
1. **监控**：每个Sentinel每秒向主、从节点和其他Sentinel发送PING命令
2. **主观下线**：如果Sentinel在`down-after-milliseconds`内没收到有效回复，标记节点为主观下线
3. **客观下线**：当足够数量（quorum）的Sentinel都认为节点主观下线，标记为客观下线
4. **选举领导者**：所有Sentinel通过Raft算法选举出一个领导者负责故障转移
5. **故障转移**：
   - 领导者Sentinel从从节点中选择一个作为新主节点
   - 向其他从节点发送`SLAVEOF`命令
   - 通知客户端主节点已变更

### 9. Redis Cluster 的架构和原理？
**架构特点**：
- 去中心化架构，无代理节点
- 16384个哈希槽，每个节点负责一部分槽
- 自动数据分片，支持在线水平扩展
- 内置主从复制和高可用

**数据分片原理**：
1. **槽分配**：集群将整个数据集分为16384个槽
2. **键路由**：对每个key计算CRC16，然后对16384取模得到槽位
3. **重定向**：客户端连接到错误的节点时，会收到MOVED/ASK重定向
4. **智能客户端**：优秀的客户端会缓存槽位到节点的映射

**故障检测与恢复**：
1. 节点间通过Gossip协议通信
2. 半数以上主节点认为某节点不可达，则标记为故障
3. 如果主节点故障，其从节点会发起选举成为新主节点
4. 集群需要大多数主节点存活才能正常工作

## 四、高级特性与使用

### 10. Redis 的事务机制与ACID属性？
**事务命令**：
- `MULTI`：开启事务
- `EXEC`：执行事务
- `DISCARD`：取消事务
- `WATCH`：乐观锁，监视key的变化
- `UNWATCH`：取消所有监视

**ACID属性**：
- **原子性**：事务中的命令要么全部执行，要么全部不执行
- **一致性**：事务执行前后，数据库完整性不被破坏
- **隔离性**：Redis单线程执行，自然保证隔离性
- **持久性**：取决于持久化配置，不保证强持久性

**注意**：Redis事务不支持回滚，如果某个命令失败，其他命令仍会执行。

### 11. Redis 如何实现分布式锁？
**基本实现**：

**RedLock算法**（分布式环境）：
1. 获取当前时间
2. 依次尝试从N个独立节点获取锁
3. 计算获取锁的总耗时，如果小于锁的超时时间，并且从大多数节点获取成功，则认为获取锁成功
4. 锁的实际有效时间 = 初始有效时间 - 获取锁的总耗时
5. 如果获取锁失败，向所有节点发送释放锁请求

**注意事项**：
- 设置合理的超时时间
- 使用唯一值作为value，避免误删
- 考虑锁的续期问题
- 评估时钟漂移对RedLock的影响

### 12. 缓存穿透、击穿、雪崩问题及解决方案
**缓存穿透**
- **问题**：查询不存在的数据，请求绕过缓存直接访问数据库
- **解决方案**：
  1. 缓存空对象：对不存在的数据也缓存，设置较短的过期时间
  2. 布隆过滤器：在缓存前增加布隆过滤器，判断数据是否存在
  3. 接口校验：对请求参数进行合法性校验

**缓存击穿**
- **问题**：热点key过期瞬间，大量请求直接访问数据库
- **解决方案**：
  1. 热点key永不过期：定期异步刷新
  2. 互斥锁：使用分布式锁，只允许一个线程重建缓存
  3. 逻辑过期：设置逻辑过期时间，实际缓存不过期

**缓存雪崩**
- **问题**：大量key同时过期或缓存服务宕机
- **解决方案**：
  1. 过期时间随机化：在基础过期时间上增加随机值
  2. 缓存永不过期+后台更新：实际过期时间远大于业务过期时间
  3. 多级缓存：本地缓存+分布式缓存
  4. 熔断降级：服务熔断，快速失败
  5. 集群部署：主从+哨兵或集群模式保证高可用

## 五、性能优化与监控

### 13. Redis 常见性能问题及优化
**常见问题**：
1. **慢查询**
   - 原因：keys *、hgetall、大key操作
   - 优化：使用scan、分批获取、拆分大key
2. **CPU使用率高**
   - 原因：复杂命令、频繁持久化、连接数过多
   - 优化：避免O(N)命令、调整持久化策略、连接池
3. **内存使用率高**
   - 原因：大key、内存碎片、无淘汰策略
   - 优化：数据分片、内存压缩、合理淘汰策略
4. **网络瓶颈**
   - 原因：频繁小数据量请求、跨机房访问
   - 优化：pipeline、mget/mset、就近部署

**优化建议**：
- 使用pipeline减少网络往返
- 使用连接池管理连接
- 避免大key和hotkey
- 合理配置持久化策略
- 监控内存碎片率，定期重启

### 14. Redis 监控指标有哪些？
**关键指标**：
1. **性能指标**
   - QPS：每秒查询数
   - 延迟：P50/P95/P99/P999延迟
   - 命中率：keyspace_hits / (keyspace_hits + keyspace_misses)
2. **资源指标**
   - 内存：used_memory、mem_fragmentation_ratio
   - CPU：used_cpu_sys、used_cpu_user
   - 网络：instantaneous_input_kbps、instantaneous_output_kbps
3. **状态指标**
   - 连接数：connected_clients
   - 阻塞客户端数：blocked_clients
   - 复制状态：master_link_status、master_sync_in_progress
4. **持久化指标**
   - RDB：rdb_last_save_time、rdb_changes_since_last_save
   - AOF：aof_current_size、aof_buffer_length

**监控工具**：
- `redis-cli --stat`：实时统计
- `redis-cli --bigkeys`：查找大key
- `redis-cli --latency`：延迟测试
- `INFO`命令：获取所有信息
- 可视化工具：RedisInsight、Redis Manager

## 六、运维与实践

### 16. 如何保证缓存与数据库的数据一致性？
**缓存策略**：
1. **Cache Aside（旁路缓存）**：
   - 读：先读缓存，命中返回，未命中读数据库并写入缓存
   - 写：更新数据库，删除缓存
   - 问题：先更新数据库再删除缓存的时序问题
2. **Read/Write Through**：
   - 读：缓存存在则返回，不存在由缓存负责从数据库加载
   - 写：先更新缓存，缓存同步更新数据库
3. **Write Behind**：
   - 更新缓存，异步批量更新数据库
   - 性能最好，但可能丢失数据

**数据一致性解决方案**：
1. **延迟双删**：更新数据库后，删除缓存，延迟一段时间再删一次
2. **订阅数据库binlog**：通过Canal等工具订阅数据库变更，更新缓存
3. **设置合理的过期时间**：允许短暂的不一致，最终一致
4. **加分布式锁**：保证操作的顺序性

**最佳实践**：
- 读多写少的场景用Cache Aside
- 一致性要求高的场景结合消息队列
- 设置合理的缓存过期时间
- 对实时性要求高的数据不缓存

### 17. Redis 大key和热key问题如何处理？
**大key问题**：
- **定义**：string类型>10KB，hash/set/zset/list等元素>5000个
- **风险**：阻塞请求、内存不均、网络拥塞、迁移困难
- **解决方案**：
  1. 拆分：将大key拆分为多个小key
  2. 压缩：value压缩存储
  3. 删除优化：使用unlink异步删除
  4. 定期清理：监控并清理大key

**热key问题**：
- **定义**：访问频率高的key
- **风险**：单节点压力大，可能成为性能瓶颈
- **解决方案**：
  1. 本地缓存：在应用层做本地缓存
  2. 多级缓存：本地+分布式缓存
  3. 读写分离：主节点写，多个从节点读
  4. 数据分片：将热key拆分为多个key
  5. 随机过期：避免同时过期造成缓存击穿

### 18. Redis 集群扩缩容步骤
**扩容步骤**：
1. 准备新节点，配置并启动
2. 将新节点加入集群：`redis-cli --cluster add-node new_ip:port existing_ip:port`
3. 重新分片：`redis-cli --cluster reshard existing_ip:port`
4. 迁移槽位：指定源节点、目标节点和槽位数量
5. 添加从节点：`redis-cli --cluster add-node --slave --master-id master_id slave_ip:port existing_ip:port`

**缩容步骤**：
1. 迁移待删除节点的槽位到其他节点
2. 删除空节点：`redis-cli --cluster del-node ip:port node_id`
3. 验证集群状态

**注意事项**：
- 业务低峰期操作
- 提前规划槽位迁移计划
- 监控迁移过程中的性能
- 验证数据一致性
- 更新客户端配置
