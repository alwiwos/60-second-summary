**RedLock算法**（分布式环境）：
1. 获取当前时间
2. 依次尝试从N个独立节点获取锁
3. 计算获取锁的总耗时，如果小于锁的超时时间，并且从大多数节点获取成功，则认为获取锁成功
4. 锁的实际有效时间 = 初始有效时间 - 获取锁的总耗时
5. 如果获取锁失败，向所有节点发送释放锁请求

**注意事项**：
- 设置合理的超时时间
- 使用唯一值作为value，避免误删
- 考虑锁的续期问题
- 评估时钟漂移对RedLock的影响

### 12. 缓存穿透、击穿、雪崩问题及解决方案
**缓存穿透**
- **问题**：查询不存在的数据，请求绕过缓存直接访问数据库
- **解决方案**：
  1. 缓存空对象：对不存在的数据也缓存，设置较短的过期时间
  2. 布隆过滤器：在缓存前增加布隆过滤器，判断数据是否存在
  3. 接口校验：对请求参数进行合法性校验

**缓存击穿**
- **问题**：热点key过期瞬间，大量请求直接访问数据库
- **解决方案**：
  1. 热点key永不过期：定期异步刷新
  2. 互斥锁：使用分布式锁，只允许一个线程重建缓存
  3. 逻辑过期：设置逻辑过期时间，实际缓存不过期

**缓存雪崩**
- **问题**：大量key同时过期或缓存服务宕机
- **解决方案**：
  1. 过期时间随机化：在基础过期时间上增加随机值
  2. 缓存永不过期+后台更新：实际过期时间远大于业务过期时间
  3. 多级缓存：本地缓存+分布式缓存
  4. 熔断降级：服务熔断，快速失败
  5. 集群部署：主从+哨兵或集群模式保证高可用

## 六、运维与实践

### 16. 如何保证缓存与数据库的数据一致性？
**缓存策略**：
1. **Cache Aside（旁路缓存）**：
   - 读：先读缓存，命中返回，未命中读数据库并写入缓存
   - 写：更新数据库，删除缓存
   - 问题：先更新数据库再删除缓存的时序问题
2. **Read/Write Through**：
   - 读：缓存存在则返回，不存在由缓存负责从数据库加载
   - 写：先更新缓存，缓存同步更新数据库
3. **Write Behind**：
   - 更新缓存，异步批量更新数据库
   - 性能最好，但可能丢失数据

**数据一致性解决方案**：
1. **延迟双删**：更新数据库后，删除缓存，延迟一段时间再删一次
2. **订阅数据库binlog**：通过Canal等工具订阅数据库变更，更新缓存
3. **设置合理的过期时间**：允许短暂的不一致，最终一致
4. **加分布式锁**：保证操作的顺序性

**最佳实践**：
- 读多写少的场景用Cache Aside
- 一致性要求高的场景结合消息队列
- 设置合理的缓存过期时间
- 对实时性要求高的数据不缓存

### 17. Redis 大key和热key问题如何处理？
**大key问题**：
- **定义**：string类型>10KB，hash/set/zset/list等元素>5000个
- **风险**：阻塞请求、内存不均、网络拥塞、迁移困难
- **解决方案**：
  1. 拆分：将大key拆分为多个小key
  2. 压缩：value压缩存储
  3. 删除优化：使用unlink异步删除
  4. 定期清理：监控并清理大key

**热key问题**：
- **定义**：访问频率高的key
- **风险**：单节点压力大，可能成为性能瓶颈
- **解决方案**：
  1. 本地缓存：在应用层做本地缓存
  2. 多级缓存：本地+分布式缓存
  3. 读写分离：主节点写，多个从节点读
  4. 数据分片：将热key拆分为多个key
  5. 随机过期：避免同时过期造成缓存击穿

### 18. Redis 集群扩缩容步骤
**扩容步骤**：
1. 准备新节点，配置并启动
2. 将新节点加入集群：`redis-cli --cluster add-node new_ip:port existing_ip:port`
3. 重新分片：`redis-cli --cluster reshard existing_ip:port`
4. 迁移槽位：指定源节点、目标节点和槽位数量
5. 添加从节点：`redis-cli --cluster add-node --slave --master-id master_id slave_ip:port existing_ip:port`

**缩容步骤**：
1. 迁移待删除节点的槽位到其他节点
2. 删除空节点：`redis-cli --cluster del-node ip:port node_id`
3. 验证集群状态

**注意事项**：
- 业务低峰期操作
- 提前规划槽位迁移计划
- 监控迁移过程中的性能
- 验证数据一致性
- 更新客户端配置
