# Kafka 常见面试题及答案

## 一、基础概念与架构

### 1. Kafka 的核心组件及其作用？
- **Producer（生产者）**：应用程序，负责创建并发送消息到指定的 Topic。
- **Broker（代理服务器）**：Kafka 集群中的单个服务节点，负责消息的接收、存储和提供查询服务。一个集群由多个 Broker 组成。
- **Consumer（消费者）**：从 Topic 订阅并消费消息的应用程序。
- **Topic（主题）**：消息的逻辑分类，生产者向其发布消息，消费者从其订阅消息。
- **Partition（分区）**：每个 Topic 被划分为一个或多个 Partition，它是物理上的概念。分区提供了并行处理的能力，是 Kafka 实现高吞吐量的基础。
- **Replica（副本）**：每个 Partition 可以有多个副本，用于提供数据冗余和高可用。副本分为 **Leader** 和 **Follower**，所有读写请求都由 Leader 副本处理。
- **Consumer Group（消费者组）**：由多个 Consumer 实例组成，共同消费一个或多个 Topic 的消息。组内的消费者实例会瓜分订阅 Topic 的所有分区，实现负载均衡。

### 2. 为什么要设计分区（Partition）？
1.  **并行处理与水平扩展**：不同分区可以分布在集群的不同 Broker 上，允许生产者并发地向多个分区写入，消费者并发地从多个分区读取，极大提升了系统的吞吐量。
2.  **负载均衡**：分区数据可以均匀分布在集群中，避免单个 Broker 成为性能瓶颈。
3.  **保障消息顺序性**：Kafka 只能保证在单个分区内的消息是有序的（先进先出）。这是分区与顺序性之间的权衡。

### 3. 什么是副本（Replica）和 ISR？
- **副本**：是分区的备份，用于防止数据丢失。`replication.factor=N` 表示每个分区有 N 个副本。
- **Leader Replica**：每个分区在某一时刻只有一个 Leader，负责处理所有的读写请求。
- **Follower Replica**：被动地、异步地从 Leader 复制数据，作为热备份。
- **ISR (In-Sync Replicas)**：指当前与 Leader 保持同步的副本集合（包括 Leader 自己）。一个 Follower 是否在 ISR 中，取决于它是否在 `replica.lag.time.max.ms`（默认 30 秒）时间内追上了 Leader 的进度。

### 4. Kafka 如何保证高可用？
主要通过 **多副本机制** 和 **Leader 选举** 实现：
1.  数据冗余：一个分区的多个副本分布在不同 Broker 上。
2.  Leader 故障转移：当某个分区的 Leader 副本所在的 Broker 宕机时，Kafka Controller 会从该分区的 ISR 集合中选举出一个新的 Leader，继续对外提供服务。这个过程对用户透明。
3.  通过 `unclean.leader.election.enable=false` 可以确保只有 ISR 中的副本才能被选为 Leader，避免了数据丢失，但可能牺牲可用性。

## 二、可靠性保障

### 5. 如何保证消息不丢失？
需要从生产者、Broker、消费者三端协同配置。

- **生产者端**：
    - 设置 `acks=all`（或 `acks=-1`）。这意味着 Leader 必须等待 ISR 中所有副本都成功写入消息，才会向生产者发送成功响应。这是最强的持久化保证。
    - 设置 `retries` 为一个较大的值（如 `Integer.MAX_VALUE`），并配合 `retry.backoff.ms` 在发送失败时自动重试。
    - 避免单次发送消息过大，以防网络传输失败率增高。

- **Broker端**：
    - 设置 `replication.factor >= 3`，确保每个分区有足够副本。
    - 设置 `min.insync.replicas > 1`（例如 2）。这表示当生产者设置 `acks=all` 时，Leader 认为“写入成功”所需的最少 ISR 副本数。即使一个副本挂掉，数据也不会丢失。
    - 设置 `unclean.leader.election.enable=false`，禁止从非 ISR 集合中选举 Leader。

- **消费者端**：
    - 关闭自动提交 `enable.auto.commit=false`。
    - 在业务逻辑处理完成后，**手动提交偏移量（Offset）**。这样可以避免消息被消费后但未处理完，因消费者崩溃导致 Offset 已提交，消息被视为“已消费”而丢失的情况。

### 6. 如何避免消息重复消费？
消息重复通常发生在“至少一次”的投递语义下。解决方案主要是 **消费端实现幂等性**。
1.  **根本原因**：消费者处理完消息后，在提交 Offset 前崩溃，重启后会从上一次提交的 Offset 处重新消费，导致重复。
2.  **解决方案**：
    - **业务逻辑幂等设计**：这是最通用的方法。例如，消费消息时先根据消息唯一ID（如订单号）查询数据库，判断该消息是否已处理过。
    - **利用 Kafka 的幂等生产者和事务**：这主要用于解决生产者端的消息重复（即“精确一次”的发送语义），对于消费端的重复，仍需业务幂等来保证。

### 7. Kafka 支持哪些消息投递语义？
- **最多一次（At Most Once）**：消息可能丢失，但绝不会重复。生产者设置 `acks=0` 或消费者先提交 Offset 后处理消息可实现。
- **至少一次（At Least Once）**：消息绝不会丢失，但可能重复。生产者设置 `acks=all` + 重试，消费者先处理消息再手动提交 Offset 可实现。这是最常用的模式。
- **精确一次（Exactly Once）**：消息有且仅被处理一次。这是最理想但最难实现的语义。Kafka 通过 **幂等性生产者（`enable.idempotence=true`）** 和 **事务 API** 在 `Kafka Streams` 或跨分区、跨主题的场景下提供支持，但消费者端仍需结合业务幂等来实现端到端的精确一次。

## 三、性能与原理

### 8. Kafka 为什么能做到高吞吐、低延迟？
1.  **顺序读写磁盘**：Kafka 将消息**追加**到 Partition 的日志文件末尾，充分利用了磁盘顺序读写速度远快于随机读写的特性（可与内存随机读写媲美）。
2.  **零拷贝技术**：消费者读取数据时，Kafka 使用 `sendfile` 系统调用，数据直接从内核的 PageCache 通过 DMA 拷贝到网卡缓冲区，避免了在用户态和内核态之间的多次数据拷贝，极大提升了性能。
3.  **页缓存（PageCache）**：读写操作直接与操作系统的页缓存交互，而不是直接读写磁盘。读操作优先从 PageCache 命中，写操作也先写入 PageCache，由操作系统异步刷盘。
4.  **批量处理**：生产者支持批量发送消息（配置 `linger.ms` 和 `batch.size`），Broker 顺序写入，消费者批量拉取，减少了网络 I/O 和磁盘 I/O 的次数。
5.  **数据压缩**：生产者可将批量消息压缩后发送，减少网络传输和磁盘存储的压力，支持 Snappy、LZ4、ZSTD 等高效压缩算法。
6.  **分区并行机制**：通过多 Partition 将数据分散到多个 Broker 和磁盘，实现了生产和消费的完全并行化。

### 9. 如何设置合理的分区数？
分区数是 Kafka 并行度的基本单位。并非越多越好。
- **考量因素**：
    1.  **吞吐目标**：单个分区在 Broker 上的写入/读取是有上限的。可通过压测估算单分区吞吐，再根据总目标吞吐量计算所需分区数。
    2.  **消费者数量**：一个分区在同一时间只能被一个消费者组内的一个消费者消费。分区数决定了消费者组的最大并行度。通常，分区数应 >= 消费者组内消费者数量。
    3.  **可用性**：更多的分区意味着更多的 Leader，选举和元数据管理的开销更大。
    4.  **集群规模**：分区总数过大（如数万）会显著增加 ZooKeeper/KRaft 和 Controller 的负担。
- **一般建议**：
    - 对于追求高吞吐的 Topic，可以从一个适中的数量开始（如 6-12），根据监控指标（如分区 Leader 分布是否均衡、消费延迟等）进行扩容。
    - 注意：分区数只能增加，不能减少。

## 四、消费者与消费组

### 10. 什么是 Consumer Group Rebalance？触发条件是什么？
- **定义**：Rebalance 是指在一个消费者组内，当消费者成员发生增加或减少，或者订阅的 Topic 分区数发生变化时，由 Coordinator 触发的一个过程，目的是重新分配分区给组内的各个消费者，以实现负载均衡。
- **触发条件**：
    1.  组内成员变化：有新的消费者加入组，或现有消费者崩溃、主动离开组。
    2.  订阅的 Topic 分区数发生变化（例如管理员增加了分区）。
    3.  订阅的 Topic 列表发生变化。
- **负面影响**：在 Rebalance 期间，**整个消费者组会停止工作**，直到分配完成。频繁的 Rebalance 会严重影响消费效率。

### 11. 如何避免不必要的 Rebalance？
1.  **调整会话超时时间**：`session.timeout.ms`（默认45秒）。消费者必须在此时间内向 Coordinator 发送心跳，否则被视为死亡。可以适当调大，但会影响故障检测速度。
2.  **调整心跳间隔**：`heartbeat.interval.ms`（默认3秒）。调小心跳间隔，让 Coordinator 更快感知消费者存活。
3.  **调整处理超时时间**：`max.poll.interval.ms`（默认5分钟）。消费者处理一批消息的最大时间。如果业务处理逻辑很重，应调大此值，避免因处理超时被误判为死亡而触发 Rebalance。
4.  **使用 Kafka 2.4+ 的增量式重平衡协议**：新版本的 `Cooperative Sticky Assignor` 分配策略，只对受影响的分区进行重平衡，而不是让整个组停止工作。

### 12. 什么是消费偏移量（Offset）？如何提交？
- **Offset**：表示消费者在某个分区上消费到的位置。它是一个不断递增的整数。
- **提交方式**：
    - **自动提交**：`enable.auto.commit=true`，定期提交。简单但可能导致重复消费或丢失。
    - **手动提交**：`enable.auto.commit=false`。
        - **同步提交**：`consumer.commitSync()`，会阻塞直到提交成功或失败。
        - **异步提交**：`consumer.commitAsync()`，不阻塞，性能更好，但失败后无重试。通常用于关闭消费者前的最后提交。

## 五、运维与监控

### 13. 如何监控 Kafka 集群健康状态？
1.  **Under Replicated Partitions (URP)**：未充分复制的分区数。理想情况下应为0。持续大于0可能意味着有 Broker 故障或网络问题。
2.  **ISR 收缩与扩张**：监控 ISR 集合的变化。
3.  **活跃控制器数量**：应为1。如果频繁切换，说明 Controller 节点不稳定。
4.  **请求处理时间**：监控 `RequestHandlerAvgIdlePercent`，过低表示 Broker 请求处理线程池繁忙。
5.  **网络与磁盘 I/O**：基础的 Broker 资源监控。
6.  **消费延迟 (Consumer Lag)**：监控各消费者组落后生产者的消息数量，这是最关键的业务指标之一。

### 14. 如何处理消息积压（Lag）？
1.  **紧急扩容**：临时增加消费者组内的消费者实例数（但不超过分区总数）。
2.  **优化消费逻辑**：检查消费者业务代码是否存在性能瓶颈（如慢SQL、复杂计算），进行优化。
3.  **调整消费参数**：适当增大 `fetch.min.bytes` 或调整 `max.poll.records`，提高单次拉取效率。
4.  **限流降级**：对于非核心业务，可以暂时跳过或降级处理。
5.  **长远规划**：如果积压是持续的，应考虑对 Topic 进行分区扩容（需要新建 Topic 并迁移数据），或提升下游处理系统的能力。
